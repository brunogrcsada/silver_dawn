1. Organisation Goals

The client business, Silver Dawn coaches, manages coach holiday operations in the UK. The client business stores information about each customer, bookings and every coach trip. A typical trip usually has multiple retired couples, which means that more than one person is included in most bookings. Silver Dawn is a small business, so Mike Gallagher, the managing director, doesn’t need to worry about scaling the database in the future. For the first phase of the project, the main goal of introducing a database is to help staff enter customer information. This set of data is only available internally, which means that customers are not able to complete online bookings. However, phase two suggests that customers should also be able to book online, therefore the database must be designed with both customer and staff usability in mind.

2. Customer Requirements

Mike Gallagher requires a database for Silver Dawn’s coach holiday operations. This relational database must be normalised and organised so that Mike introduce online booking to Silver Dawn’s customers in the future. Mike wants to ensure that this database allows Silver Dawn staff to store and update any data related to their customers. This includes:

  • Being able to create a new customer and insert any data related to them, including their name, address/contact information and special requirements for any trip.
  • Allowing staff to create new trips based on the driver information, destination and hotel name (depending on the duration of the trip)
  • Storing business metadata including coach identification (such as its registration) and the number of seats available (This will help Silver Dawn identify whether more customers are able to book a seat in a trip).
  • Being able to organise any trips created into the destinations stated in the database. This is one of the main reasons why Mike requires a relational database; data from the destination entity should be accessed by the trip entity.

3. Database Aims

Silver Dawn requires a secure, stable and organised database which complies with legal standards. The database structure should be normalised in third normal form (minimum requirement) in order to reduce the amount of redundant data and repeats, which can cause low cardinality in certain attributes. Furthermore, the database should be able to:

  • Hold all of the necessary data with its matching data type, ensuring that staff does not enter data in any incorrect format when creating a new customer, booking, destination or trip. This includes validation checks according to data ranges and presence.
  • Prevent deletion and update of data from decreasing the database’s data integrity by ensuring the consistency of database relationships.

Within the database, all entities and attributes should be named suitably according to the data which they are storing (follow naming conventions). For example, the entity which stores all information about the customers should be named customer.

4. Project Constraints

The first phase of this project must be completed in 42 days. This includes the design of the database system with an appropriate form-based method which allows Silver Dawn’s staff to query and update data in the database. Furthermore, in order for Mike to use the database in phase two, it must be both normalised and organised; ready to store a reasonable amount of data without insertion and update anomalies.

In the first phase of the project, Mike has to ensure that staff have access to a computer system which is able to connect to the database. Unlike phase two, which will require a dedicated server in order for both customers and staff to query and insert data, the implementation of a simple database system for staff to use can be run locally without requiring extra hardware (which can increase the overall cost of the project). To allow Silver Dawn’s staff to access the database, Mike can provide them with either an inexpensive smartphone/tablet, or a dedicated desktop machine/laptop.

To design and develop a relational database, including the configuration of a local RDBMS and digital forms for staff, the hourly cost of this service will be a fixed amount of £50 p/h; progress on the project will be recorded weekly through documentation, to ensure that payment is relative to the workload being described.

5. Legal & Ethical Considerations

GDPR, being a fundamental alteration in how different businesses manage and control user data, should be taken into consideration when Silver Dawn utilises this database to store and query information about their customers and staff. The main aim of this regulation is to provide EU citizens with protection from personal data breach and misuse. Silver Dawn needs to ask their customers for consent to store any of their data in the database. The designer and implementer of the database system will not be held accountable for any illegal misuse of data if Silver Dawn does not request the correct permissions from both their staff and any other party which stores their data in Silver Dawn's servers. The business should also be able to provide their customers with instant access to their data, if they wish to do so.

While Silver Dawn is unlikely to benefit from potential investors in the future (without scaling), Mike should also take the Sarbanes-Oxley Act of 2002 into consideration. The business should ensure that staff are recordkeeping any earnings and financial reports periodically. This U.S. Congress law aims to defend possible business investors from dishonest economic reporting. The data recorded in this database should be used by Silver Dawn as a reference point when documenting their finances.

Silver Dawn should also ensure that all of their intellectual property is not stolen by another company or entity by shielding it using a copyright, design or patent protection. Regarding the data stored in the database, including customer and staff information, Silver Dawn must ensure that no data is shared with an entity outside of the organisation, including a competing brand or a social media analytics company. By sharing information without consent, you are infringing GDPR and other data protection laws such as the DPA.
